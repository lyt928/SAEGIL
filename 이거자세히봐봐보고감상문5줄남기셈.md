# SAEGIL

> **AI 기반 공사현장 안전 모니터링 및 실시간 경보 시스템**  
> 실시간 영상 분석을 통해 작업자 안전 상태를 감지하고, 위험 상황 발생 시 **이벤트 로그**, **경고**, **이벤트 시점 클립/스냅샷 저장**, **모니터링 화면 제공**까지 연결하는 프로젝트입니다.

---

## 1. 프로젝트 한눈에 보기

건설 현장은 추락, 협착, 낙하물, 장비 충돌, 보호장비 미착용 등 다양한 위험 요소가 상시 존재합니다. 기존 안전관리는 CCTV 수동 확인과 현장 순회 점검에 의존하는 경우가 많아, **실시간 대응**, **사고 전조 탐지**, **근거 기록 관리**에 한계가 있습니다.

SAEGIL은 이 문제를 해결하기 위해 다음을 목표로 합니다.

- 작업자 실시간 탐지
- PPE(안전모 등) 착용 여부 판별
- 위험구역 진입/이탈 감지
- 위험상황 조기 판단 및 경고
- 이벤트 로그 기록
- 이벤트 발생 시점의 스냅샷/영상 클립 저장
- 대시보드/웹을 통한 실시간 모니터링

즉, 이 프로젝트는 단순 객체 탐지 데모가 아니라,

**"실시간으로 보고 → 위험을 판단하고 → 기록과 근거를 남기는 현장 안전관제 시스템"**

으로 발전시키는 것을 목표로 합니다.

## 4. 핵심 설계 원칙

이 프로젝트는 아래 원칙을 중심으로 진행합니다.

### 4.1 이벤트는 프레임마다 저장하지 않는다

이벤트는 **상태가 바뀌는 시점만 기록**합니다.

예시:

- 계속 안전모가 없는 상태 → 계속 저장하지 않음
- 안전모를 쓰다가 벗은 순간 → `helmet_missing`
- 다시 착용한 순간 → `helmet_restored`
- 위험구역 밖에서 안으로 들어간 순간 → `zone_intrusion`
- 위험구역 안에서 밖으로 나온 순간 → `zone_exit`

이 원칙이 없으면 로그가 과도하게 쌓이고, 영상 클립도 중복 생성되어 운영이 매우 지저분해집니다.

### 4.2 로그만 남기지 말고 근거도 남긴다

이벤트 텍스트만 저장하면 나중에 “실제 화면에서 무슨 일이 있었는지” 확인하기 어렵습니다.

그래서 SAEGIL은 이벤트 발생 시 다음을 함께 저장하는 방향을 채택합니다.

- 이벤트 로그
- 이벤트 시점 스냅샷 이미지
- 이벤트 전후 짧은 영상 클립

### 4.3 상시 전체 녹화보다 이벤트 기반 클립 저장을 우선한다

1차 구현에서는 24시간 전체 녹화보다 아래 구조를 우선합니다.

- 평소: 실시간 분석 + 실시간 스트리밍
- 이벤트 발생 시: 전후 몇 초 클립 저장

예:

- 이벤트 발생 10초 전
- 이벤트 발생 10초 후
- 총 20초 클립 저장

이 방식이 저장 공간, 운영 복잡도, 검토 효율 측면에서 가장 현실적입니다.

### 4.4 엣지 디바이스와 서버 역할을 분리한다

젯슨 계열 장비는 현장 엣지 추론용으로 사용하고,
백엔드 서버는 저장/조회/대시보드 역할을 맡는 구조를 권장합니다.

## 5. 주요 기능 정의

### 5.1 작업자 탐지

카메라 프레임에서 작업자(person)를 탐지합니다.

### 5.2 PPE 판별

작업자와 보호장비(예: 안전모) 위치 관계를 바탕으로 착용 여부를 판정합니다.

### 5.3 위험구역 진입/이탈 감지

사전에 지정한 polygon/rectangle 위험구역에 작업자가 진입하거나 이탈하는 순간을 감지합니다.

### 5.4 위험상황 판단

초기에는 규칙 기반으로 판단합니다.

예:

- 안전모 미착용 상태
- 위험구역 내 작업자 존재
- 특정 장비 작업 반경 내 작업자 접근

### 5.5 실시간 스트리밍

현장 카메라 영상을 휴대폰/컴퓨터에서 CCTV처럼 실시간 확인할 수 있도록 웹 기반 스트리밍을 제공합니다.

### 5.6 이벤트 로그 저장

이벤트는 DB에 저장되며, 추후 필터 조회/상세 조회/API 제공이 가능해야 합니다.

### 5.7 이벤트 시점 근거 저장

이벤트 발생 시점의 스냅샷 또는 영상 클립 경로를 함께 저장합니다.

### 5.8 경고 시스템 연동

화면 오버레이, 경고 문구, 상태 표시, 경고음/부저 등과 연동합니다.

## 8. 이벤트 정의

1차 이벤트 목록은 아래 7개를 기준으로 합니다.

| 이벤트 이름 | 코드값 | 설명 |
|---|---|---|
| 안전모 미착용 | `helmet_missing` | 작업자가 안전모를 착용하지 않은 상태로 변경됨 |
| 안전모 재착용 | `helmet_restored` | 안전모 미착용 상태에서 다시 착용 상태로 복구됨 |
| 위험구역 진입 | `zone_intrusion` | 작업자가 위험구역 내부로 진입함 |
| 위험구역 이탈 | `zone_exit` | 작업자가 위험구역 밖으로 나감 |
| 경고 발생 | `alert_triggered` | 특정 위험 조건에 의해 경고가 발생함 |
| 카메라 상태 변경 | `camera_status_changed` | 카메라 연결 상태가 바뀜 |
| 모델 상태 변경 | `model_status_changed` | 모델 준비/실행/오류 상태가 바뀜 |

### 상태(status) 예시

카메라/모델은 단순 true/false보다 상태값으로 관리하는 것을 권장합니다.

- camera: `connected`, `disconnected`
- model: `ready`, `running`, `failed`

## 9. DB 설계 가이드

### 9.1 왜 PostgreSQL인가

초기 논의 결과, 이벤트 로그 시스템은 SQLite보다 PostgreSQL 기반으로 가는 방향을 우선 채택합니다.

이유:

- 정석적인 서버형 SQL 구조 학습
- 나중에 확장 쉬움
- 조회 API, 필터링, 운영 측면에서 유리

### 9.2 초기 `events` 테이블 권장 컬럼

| 컬럼명 | 설명 |
|---|---|
| `id` | PK |
| `event_type` | 이벤트 코드값 |
| `message` | 사람이 읽기 쉬운 메시지 |
| `camera_id` | 카메라 식별자 |
| `worker_id` | 작업자 식별자 |
| `zone_id` | 위험구역 식별자 |
| `frame_id` | 프레임 번호 |
| `confidence` | 모델 신뢰도 |
| `bbox` | 감지 영역 정보 |
| `status` | camera/model 상태값 |
| `snapshot_path` | 스냅샷 파일 경로 |
| `clip_path` | 영상 클립 경로 |
| `clip_start_at` | 클립 시작 시각 |
| `clip_end_at` | 클립 종료 시각 |
| `created_at` | 이벤트 발생 시각 |

### 9.3 백엔드 API 초안

- `POST /events`
- `GET /events/recent`
- `GET /events`
- `GET /events/{id}`

추가 필터 예시:

- `event_type`
- `camera_id`
- `status`
- `date range`

## 10. 실시간 스트리밍 + 이벤트 클립 저장 구조

이 프로젝트의 핵심은 **실시간 송출 기능**과 **이벤트 시점 근거 저장 기능**을 하나의 파이프라인으로 연결하는 것입니다.

### 10.1 개념 요약

같은 카메라 프레임을 동시에 세 군데에 사용합니다.

1. **AI 분석용**
2. **실시간 송출용**
3. **최근 프레임 버퍼 저장용**

그리고 이벤트가 발생하면,
최근 버퍼와 이후 프레임을 합쳐서 클립을 저장합니다.

### 10.2 작동 흐름

```text
[카메라]
   ↓
[프레임 수집]
   ↓
 ┌───────────────┬───────────────┬────────────────┐
 ↓               ↓               ↓
[AI 분석]      [실시간 송출]    [순환 버퍼 저장]
 ↓                               ↓
[상태 변화 감지]                 [최근 N초 유지]
 ↓
[이벤트 발생]
 ↓
[버퍼의 과거 프레임 + 이후 프레임 결합]
 ↓
[클립 파일 저장]
 ↓
[events 테이블에 clip_path 기록]
```

### 10.3 실시간 송출 역할

- 현장 상황을 휴대폰/컴퓨터에서 확인
- 현재 감지 결과를 UI에서 확인
- 관리자 관제용 화면 제공

### 10.4 클립 저장 역할

- 이벤트 당시 장면을 근거로 남김
- 오탐/정탐 검증 가능
- 보고서/사후 검토 가능

### 10.5 권장 저장 방식

- 평시 전체 녹화보다 이벤트 기반 저장 우선
- 예: 전 10초 + 후 10초 = 20초 클립
- 1차 구현은 스냅샷 저장부터 시작해도 됨

## 17. 초기 실행 목표(데모 기준)

다음 조건을 만족하면 1차 데모가 완성된 것으로 봅니다.

- 카메라 영상이 들어온다
- 작업자가 검출된다
- 안전모 미착용 여부를 판정한다
- 위험구역 진입 이벤트가 발생한다
- 이벤트가 DB에 저장된다
- 스냅샷이 저장된다
- 실시간 화면을 웹에서 볼 수 있다
- 이벤트를 눌렀을 때 관련 근거 화면을 확인할 수 있다

## 18. 추가 피드백

### 꼭 유지해야 할 판단

1. **이벤트 구조를 먼저 통일할 것**
2. **상태 변화 시점만 저장할 것**
3. **로그만 남기지 말고 근거를 남길 것**
4. **젯슨에 모든 역할을 몰아넣지 말 것**
5. **1차는 단일 카메라/기본 탐지부터 확실히 만들 것**

### 팀이 헷갈리기 쉬운 포인트

- 객체 탐지와 PPE 판정은 같은 게 아님
- 실시간 송출과 이벤트 클립 저장은 다른 기능이지만 같은 프레임 파이프라인에서 함께 동작해야 함
- DB 저장과 파일 저장은 별도 관리 대상임
- Jetson에서 바로 모든 대시보드 기능까지 처리하려고 하면 개발이 꼬일 가능성이 큼

### 추천 개발 순서 한 줄 요약

**카메라 입력 → 탐지 → PPE/위험구역 판정 → 상태 변화 이벤트 → DB 저장 → 스냅샷/클립 저장 → 실시간 송출 → 대시보드 연결**

---

## 19. 최종 한 줄 요약

SAEGIL은 공사현장 카메라 영상을 기반으로 작업자와 PPE 상태를 실시간 분석하고, 위험구역 및 위험상황을 상태 변화 기반 이벤트로 기록하며, 이벤트 시점의 스냅샷/클립과 함께 웹 대시보드에서 확인할 수 있도록 설계된 AI 기반 안전관제 시스템입니다.
